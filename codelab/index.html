2 · 개념

SDUI는 “화면 결정권”을 서버와 나누는 방식입니다

SDUI는 Server-Driven UI의 줄임말입니다. 서버가 데이터만 보내는 대신, 어떤 UI를 어떤 순서로 보여줄지에 대한 설명도 함께 보냅니다.

일반 앱과 비교하면 쉽습니다

일반 Android 앱
서버 데이터→앱에 고정된 UI 코드→화면

서버가 같은 데이터를 보내면 앱에 이미 배포된 Compose 코드가 화면을 결정합니다.

Server-Driven UI
UI 설명 + 데이터→앱의 Renderer→화면

서버가 목록·텍스트·순서 같은 설명을 바꾸면 앱 재배포 없이 화면 구성이 달라질 수 있습니다.

언제 유용할까요?

  • 앱 배포 주기와 별개로 프로모션, 설문, 체크리스트 구성을 바꾸고 싶을 때
  • Android가 허용한 UI와 액션 범위 안에서 서버가 화면 순서를 결정하게 할 때
  • 여러 앱 버전이 같은 문서를 안전하게 재생하도록 호환성 규칙을 운영할 수 있을 때

3 · 실행

서버를 켜고 앱을 연결해 봅니다

이 저장소는 이해를 돕기 위해 Ktor/JVM 서버와 Android 앱을 함께 제공합니다. Ktor는 Remote Compose의 필수 요소가 아니라, 문서 바이트를 전달하기 위해 이 POC가 선택한 서버입니다.

1. 서버 실행

server/는 독립 Gradle 프로젝트입니다. 아래 명령은 Android 앱을 빌드하지 않고 서버만 실행합니다.

저장소 POC
cd samples/remote-state-lab/server
./gradlew run
i
127.0.0.1과 10.0.2.2는 왜 다를까요?

127.0.0.1은 그 주소를 사용하는 기기 자신입니다. 에뮬레이터 안의 127.0.0.1은 Mac이 아니므로, Android Emulator가 host 컴퓨터를 가리키도록 제공하는 10.0.2.2를 씁니다.

2. 앱 실행과 연결

저장소 POC
cd samples/remote-state-lab
./gradlew :app:assembleDebug
  1. 1
    앱을 에뮬레이터에서 실행

    처음 보이는 화면은 일반 Jetpack Compose로 만든 연결 화면입니다.

  2. 2
    http://10.0.2.2:8080 입력

    Android Emulator에서 Mac의 8080 포트로 접근하는 주소입니다.

  3. 3
    연결 선택

    앱이 GET /document로 바이트를 받고 Remote Compose 화면을 엽니다.

관찰할 것

!
Address already in use가 나오면

이미 8080 포트를 쓰는 서버가 있다는 뜻입니다. lsof -nP -iTCP:8080 -sTCP:LISTEN으로 확인하고 기존 POC 서버를 재사용하거나 종료합니다.

4 · UI

서버 코드에서 첫 Remote UI를 작성합니다

이 코드랩에서는 Ktor/JVM이 RcScope DSL을 실행해 UI operation을 문서에 기록합니다. 이 코드는 서버에서 실행되며 Android 화면을 직접 그리지 않습니다.

이름은 비슷해도 실행 장소가 다릅니다

하려는 일Android의 Jetpack Compose서버의 RcScope DSL
세로 배치ColumnColumn
문자 표시TextText
수정자androidx.compose.ui.Modifierandroidx.compose.remote.creation.dsl.Modifier
클릭Modifier.clickable { Kotlin lambda }Modifier.onClick { document action }
결과현재 Android UI tree나중에 보낼 Remote Compose operation
!
서버의 Text는 Jetpack Compose Text가 아닙니다

createRcBuffer { ... }가 제공하는 RcScope receiver의 함수입니다. 이름이 같을 뿐 @Composable도 아니고 Android UI toolkit도 아닙니다.

가장 작은 작업 목록

Restricted RcScope · 실제 POC 경로
private fun RcScope.TaskListContent() {
    Column(
        modifier = Modifier.fillMaxSize().padding(18f),
    ) {
        Text("오늘 할 일")
        Text("API 회귀 테스트")
    }
}
  1. 1
    RcScope. 확장 함수

    Remote Compose writer를 receiver로 받아 operation을 추가합니다.

  2. 2
    Column

    세로 container를 문서에 기록하고 block 안의 child operation을 이어서 기록합니다.

  3. 3
    Text(String)

    procedural DSL에는 일반 String overload가 있으므로 이 경로에서는 .rs가 필요하지 않습니다.

왜 restricted 배지가 붙나요?

alpha14에서 이 저장소가 사용하는 RcScope와 createRcBuffer는 AndroidX source상 LIBRARY_GROUP 제한 API입니다. 서버 중심 SDUI를 가장 직접적으로 실험할 수 있지만, 안정 public 계약으로 간주하면 안 됩니다.

i
이 코드랩의 선택

“공개 API를 모두 먼저 배우기”보다 “현재 샘플이 실제로 동작하는 한 경로를 끝까지 이해하기”를 우선합니다. 다른 생성 frontend는 Step 10의 심화 자료에서 비교합니다.

5 · 문서 생성

Header와 UI를 하나의 ByteArray로 만듭니다

RcScope UI만 적는 것으로는 전송이 끝나지 않습니다. 서버는 player가 해석할 문서 속성을 지정하고, UI operation과 함께 buffer로 직렬화합니다.

서버 안에서 일어나는 네 단계

1Profile지원 protocol과 platform
→
2Header크기·density·설명
→
3Root UIColumn과 child
→
4BufferByteArray

실제 서버의 문서 생성 함수

Restricted RcScope · 실제 코드 축약
fun buildChecklistDocument(snapshot: ChecklistServerSnapshot): ByteArray =
    createRcBuffer(
        rcProfile,
        RemoteComposeWriter.HTag(Header.DOC_WIDTH, 390),
        RemoteComposeWriter.HTag(Header.DOC_HEIGHT, 720),
        RemoteComposeWriter.HTag(
            Header.DOC_DENSITY_BEHAVIOR,
            CoreDocument.DENSITY_BEHAVIOR_DP,
        ),
    ) {
        TaskListContent()
    }
  1. 1
    rcProfile

    writer와 player가 어떤 Remote Compose profile을 전제로 하는지 정합니다.

  2. 2
    Header.*

    문서 viewport와 density behavior 같은 해석 조건을 기록합니다.

  3. 3
    { TaskListContent() }

    이 block이 root UI content입니다. 실행 결과는 Kotlin 함수가 아니라 layout/text operation입니다.

  4. 4
    ByteArray

    createRcBuffer가 Header와 operation을 전송 가능한 binary document로 반환합니다.

Header, root, state를 섞지 마세요

용어역할이 샘플에서 보이는 곳
Header문서 전체의 version/property framingcreateRcBuffer 인자
Root UI화면 tree의 최상위 contentbuffer block 안의 첫 layout
Remote statePlayer가 ID로 보관·변경하는 값다음 단계의 remoteNamedInteger
!
별도의 RootState라는 API를 찾지 마세요

root layout과 runtime state는 다른 개념입니다. binary format은 flat operation을 기록하고 player가 container/end operation을 해석해 UI tree를 복원합니다.

6 · 상태

문서 안의 상태는 ID와 action으로 움직입니다

서버의 Kotlin 코드는 문서를 만들 때 한 번 실행됩니다. 재생 중에는 Android Player가 문서에 기록된 state ID를 읽고 setValue action을 적용합니다.

Jetpack Compose state와 실행 방식이 다릅니다

JETPACK COMPOSE Kotlin 코드가 다시 실행됩니다

mutableStateOf가 바뀌면 Android 프로세스에서 recomposition이 일어나고 Composable을 다시 평가합니다.

≠
REMOTE DOCUMENT Player가 기록된 operation을 평가합니다

click은 서버 함수를 다시 호출하지 않습니다. state map의 ID 값을 바꾸고 StateLayout이 이미 기록된 child를 다시 선택합니다.

procedural DSL로 완료 토글 만들기

Restricted RcScope · 개념용 축약
val done = remoteNamedInteger("done", 0)

StateLayout(stateIndex = done) {
    Text(
        "미완료",
        modifier = Modifier.onClick {
            setValue(done, 1)
        },
    )
    Text(
        "완료",
        modifier = Modifier.onClick {
            setValue(done, 0)
        },
    )
}
  1. 1
    remoteNamedInteger("done", 0)

    초깃값 0을 가진 document state를 만들고 writer가 사용할 ID를 연결합니다.

  2. 2
    StateLayout(stateIndex = done)

    state 값과 같은 순서의 child 하나를 표시하는 selector operation을 기록합니다.

  3. 3
    첫 번째 child

    done = 0일 때 “미완료”를 표시하고, click action은 값을 1로 바꿉니다.

  4. 4
    두 번째 child

    done = 1일 때 “완료”를 표시하고, click action은 값을 0으로 되돌립니다.

핵심
onClick block은 임의의 서버 로직이 아닙니다

문서 생성 중 setValue action을 기록하는 DSL block입니다. 재생 중 click에서 Ktor의 이 Kotlin block이 실행되는 것이 아닙니다.

생성된 문서를 단순화하면 이렇습니다

STATENamedInt #42 = 0이름은 done
SELECTORStateLayout indexId = #42현재 값과 같은 child만 표시
CHILD 0“미완료”click → set #42 = 1
CHILD 1“완료”click → set #42 = 0

ID #42는 이해를 위한 예시입니다. 실제 ID는 writer가 문서를 만들며 할당합니다.

Player 동작을 직접 눌러봅니다

학습용 모델 · 실제 Player UI 아님Remote state playback
RemoteComposeState[#42] = 0
child 0미완료click action: #42에 1 기록
child 1완료click action: #42에 0 기록

초기 문서를 재생했습니다. #42가 0이므로 child 0만 표시합니다.

!
document state는 서버 DB가 아닙니다

Player 안에서 표시 상태만 바뀐 것입니다. 저장된 작업, 권한, 결제처럼 영속성과 domain rule이 필요한 값은 Repository와 서버가 계속 소유해야 합니다.

어떤 상태를 어디에 둘까요?

상태 예시소유자이유
현재 펼쳐진 설명, 작은 화면 selectorRemote document state문서 안에서 즉시 바뀌고 reset되어도 안전
서버 연결, loading/errorAndroid ViewModellifecycle, retry, fallback UI가 필요
사용자 입력 중인 문자열Android UI/ViewModel현재 POC의 procedural component에는 일반 IME TextField가 없음
저장된 작업, 권한, 결제 상태Repository/server영속성, 인증, domain rule이 필요

7 · 화면 전환

목록과 상세를 한 문서에 함께 기록합니다

현재 샘플은 Android Navigation을 호출하지 않습니다. screen state를 selector로 사용해 한 Remote Compose 문서 안에서 목록 child와 상세 child를 교체합니다.

실제 샘플의 StateLayout 구조

Restricted RcScope · 실제 서버 코드
val screen = remoteNamedInteger("screen", 0)

StateLayout(
    stateIndex = screen,
    modifier = Modifier.fillMaxSize(),
) {
    TaskListScreen(snapshot, screen)

    snapshot.tasks.forEach { task ->
        TaskDetailScreen(snapshot, task, screen)
    }
}
screen = 0작업 목록첫 번째 child
⇄
screen = index + 1선택한 작업 상세각 task의 child

행을 누르면 왜 상세가 보이나요?

Restricted RcScope · 실제 동작 축약
snapshot.tasks.forEachIndexed { index, task ->
    TaskRow(
        task = task,
        detailScreenIndex = index + 1,
        screen = screen,
    )
}

// TaskRow 내부
Modifier.onClick {
    setValue(screen, detailScreenIndex)
}

// 상세의 뒤로 버튼
Modifier.onClick {
    setValue(screen, 0)
}
  1. 1
    문서 생성

    서버 snapshot의 task 수만큼 상세 child를 미리 기록합니다.

  2. 2
    행 click

    Player가 screen ID에 index + 1을 기록합니다.

  3. 3
    child 선택

    StateLayout이 바뀐 index에 해당하는 상세 child를 표시합니다. UI를 위에 덮는 방식이 아닙니다.

  4. 4
    목록 복귀

    screen = 0으로 바꾸면 첫 번째 child인 목록을 다시 표시합니다.

이것은 Android Navigation인가요?

아닙니다. Activity, Fragment, Navigation Compose destination이나 back stack은 변하지 않습니다.

StateLayout이 적합

작은 설문 단계, 문서 내부 목록/상세, reset되어도 안전한 임시 전환

Android Navigation이 적합

deep link, 시스템 back, 다른 feature graph, process restore, 앱 전체 화면 소유권

8 · 앱과 연결

서버 저장이 필요하면 hostAction으로 앱에 알립니다

setValue는 document-local 상태만 바꿉니다. 입력창, API, 인증, 저장이 필요하면 문서는 이름 있는 event를 발생시키고 Android host가 실제 business 작업을 수행합니다.

task.create가 API 요청으로 이어지는 전체 과정

RcScope 문서hostAction("task.create")

문서에 named action 기록

→
Android callbackonNamedAction

Player가 이름을 host에 전달

→
ViewModelhandleHostAction

allowlist 검사와 입력 UI 요청

→
Repository/APIPOST /tasks

입력값을 Ktor 서버에 저장

서버가 문서에 기록하는 action

Restricted RcScope · 실제 서버 문법
ActionButton(label = "+  새 작업 작성") {
    hostAction("task.create")
}

ActionButton(label = "삭제") {
    hostAction("task.delete.${task.id}")
}

hostAction 자체는 HTTP 요청을 보내지 않습니다. 문자열을 문서 action으로 기록할 뿐이며, Android host가 그 이름을 해석했기 때문에 결과적으로 API 호출이 일어납니다.

Android 앱이 business action으로 번역합니다

저장소 POC · host integration
RemoteDocumentPlayer(
    document = document,
    onNamedAction = { name, value, _ ->
        viewModel.handleHostAction(name, value)
    },
)
!
문서 문자열을 URL이나 함수처럼 실행하지 마세요

task.create와 양의 정수 ID를 가진 task.delete.<id>처럼 앱이 아는 command만 허용하고 payload를 검증해야 합니다.

RemoteDocumentPlayer도 alpha14 고정 source 기준 restricted API입니다. 이 코드는 저장소 POC의 실제 경계를 보여 주며 안정 public player 계약이 아닙니다.

왜 텍스트 입력은 Android가 담당하나요?

현재 POC에서 확인한 procedural component surface에는 일반 Android TextField처럼 IME로 자유 입력을 받는 component가 없습니다. 그래서 task.create를 받으면 Android가 native 입력 UI를 열고, 입력 완료 후 Repository가 서버에 저장합니다.

결론
Remote Compose가 ViewModel이나 Repository를 대신하지 않습니다

문서는 표시·문서 내부 상태·허용된 event를 표현합니다. lifecycle, 입력, 인증, retry, persistence와 domain rule은 기존 Android architecture에 남습니다.

공식 Ktor용 Remote Compose 문서가 있나요?

현재 조사 범위에서 Remote Compose와 Ktor를 연결하는 공식 guide나 Codelab은 확인하지 못했습니다. AndroidX는 JVM 생성 artifact를 제공하지만, Ktor route와 배포 구조는 이 저장소의 engineering POC입니다.

서버가 바꾸면 앱에 무엇이 보일까요?

작업을 추가한 뒤 새 문서를 요청하면 서버는 새로운 목록으로 문서를 다시 만듭니다. 앱은 새 바이트를 받아 player에 전달하고, 사용자는 갱신된 목록을 봅니다. 이 POC의 “자동 동기화”는 이 재요청 과정을 뜻합니다.

서버 중심 SDUI 한 경로를 끝까지 연결했습니다

이 코드랩의 기준은 Ktor/JVM → RcScope document → HTTP ByteArray → Android Player입니다. 다른 생성 frontend를 섞지 않고 현재 POC의 실행 모델과 책임 경계를 배웠습니다.

한 번에 다시 보기

서버 데이터→RcScope UI operation→Remote Compose bytes→Android Player→화면
사용자 클릭→hostAction→Android ViewModel→Repository/API→새 문서 요청

다른 제작 방식은 별도 심화 과제입니다

Android 또는 Android 기반 CI에서 @RemoteComposable content를 capture하는 public Compose frontend도 존재합니다. 그 경로에서 RemoteText, .rs, .rb, valueChange가 등장합니다. 하지만 서버에서 RcScope.Text(String)를 직접 사용하는 이번 경로에는 필요하지 않습니다.
